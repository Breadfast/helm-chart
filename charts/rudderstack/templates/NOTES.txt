OPERATIONAL CAVEAT - processor failover on hard node failure:
  The processor is a StatefulSet with replicas=1 and no PDB. On a normal node
  deletion / cluster-autoscale event it reschedules automatically. But on a HARD
  node failure (node goes NotReady but the Node/Pod objects are not deleted),
  Kubernetes will NOT recreate the pod until it is certain the old one is gone -
  it can stay stuck "Terminating"/NotReady until the Node object is removed or the
  pod is force-deleted:
      kubectl -n {{ .Release.Namespace }} delete pod {{ include "processor.fullname" . }}-0 --grace-period=0 --force
  Recommended alert: "processor pod not Ready > 3-5 min" with the above as the
  runbook step. Gateways keep buffering events to Cloud SQL during this window,
  so the impact is processing/delivery latency, NOT data loss.

First boot on an empty Cloud SQL:
  gw_* tables are created by the gateway (the writer). The processor opens gw_*
  read-only and tolerates their absence (its /health Ping is connection-level),
  so it will not crash-loop before the gateway has started.
